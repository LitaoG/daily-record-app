# 同步与隐私

## 1. 原则

- 核心功能离线可用。
- Room 是客户端单一事实来源。
- 网络失败不能导致本地记录丢失或界面回滚。
- 云端按用户强制隔离，不能仅依赖客户端过滤。
- 后端实现可以替换，业务层不感知具体 SDK。

## 2. 同步流程

### 本地写入

1. 在 Room 事务中写入业务实体。
2. 同一事务创建带幂等 ID 的 Outbox 操作。
3. UI 从 Room Flow 立即更新。
4. 请求唯一 WorkManager 任务。

### 上传

1. 网络可用且用户已登录时读取待处理操作。
2. 按创建顺序或实体依赖顺序批量上传。
3. 服务端按 operation ID 去重。
4. 成功后标记确认并清理；失败使用指数退避。
5. 未授权错误暂停队列，等待重新登录，不无限重试。

### 拉取

- 按服务端修订号或更新时间游标拉取变化。
- 使用 tombstone 同步删除。
- 数据转换与 Room 写入放在事务中。
- 拉取完成后保存游标。

## 3. 冲突策略

MVP 是个人应用，多设备同时编辑同一活动同一天属于低频但必须可解释的情况。

初始建议：

- 不同记录：自然合并。
- 同一 DailyRecord：比较服务端 revision，并采用最后一次明确编辑；保留足够元数据诊断。
- Activity 归档与编辑冲突：较新的 revision 生效。
- 删除：使用 tombstone，不能只在某一端物理删除。

在实现前用两台设备模拟离线编辑，确认“最后写入”是否足够。如果用户需要无损合并次数，升级为事件型记录或增量操作模型。

## 4. 游客数据绑定

- 游客拥有稳定的本地 ownerId。
- 登录后先展示将要合并的数据数量。
- 合并为可重试的本地事务和同步任务。
- 同一设备重复执行不会复制活动或记录。
- 合并成功前不删除游客数据。

## 5. 后端决策

优先评估 Supabase/PostgreSQL，因为关系模型、统计和 RLS 较直观，并可自托管。Firebase 是备选，适合快速身份和实时同步，但不能让 Firestore 类型进入领域层。主要在中国大陆使用时，正式选择前必须实测网络可用性和部署区域。

## 6. 隐私基线

- 默认不要求登录。
- 只收集账号和同步必需信息。
- 备注和行为记录不进入分析事件或普通日志。
- 本地应用支持系统生物识别锁。
- 云端使用 TLS 和服务商静态加密。
- 每个表启用用户级服务端访问策略。
- 支持导出和删除账户数据。
- 崩溃报告在启用前完成字段审查，并允许用户了解其用途。

## 7. 端到端加密

端到端加密能防止服务端看到记录内容，但会增加密钥恢复、多设备授权、搜索和数据恢复复杂度。架构应避免阻止未来增加 E2EE，但 MVP 是否实现必须在选定后端前明确。若实现，需要单独威胁模型和恢复密钥方案，不能仅把固定密钥写在客户端。

## 8. 公开仓库配置

- 只提交 `.env.example`，不提交 `.env`。
- 服务端管理密钥只放在部署环境或 GitHub Secrets。
- Android 客户端的公开配置仍必须配合 RLS/服务端权限，不能被当作安全边界。
- Release keystore 和密码不进入仓库。
- 示例数据库和截图使用虚构数据。
