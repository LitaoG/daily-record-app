# 架构决策记录

此文件记录会影响后续实现的长期决策。状态可为 `Proposed`、`Accepted`、`Superseded`。

## ADR-001：Android 原生优先

- 状态：Accepted
- 决策：第一版使用 Kotlin 和 Jetpack Compose 开发 Android 原生应用。
- 原因：当前目标明确为 Android；原生方案与平台数据库、后台任务和生物识别集成最直接。
- 后果：如果一年内明确要开发 iOS，需要在工程创建前重新评估 KMP，而不是默认增加跨平台复杂度。

## ADR-002：通用活动模型

- 状态：Accepted
- 决策：手冲和健身作为可配置 `Activity`，不创建各自独立的数据表和 Repository。
- 原因：未来增加活动时只需要配置和局部 UI 扩展。
- 后果：计量方式和统计规则必须显式建模。

## ADR-003：离线优先与 Room 单一事实来源

- 状态：Accepted
- 决策：所有记录先写 Room，UI 只读取 Room；云端通过后台队列同步。
- 原因：保证无网可用和交互即时，也便于测试和替换后端。
- 后果：需要 Outbox、同步状态、冲突策略和迁移测试。

## ADR-004：公开仓库与 Apache 2.0

- 状态：Accepted
- 决策：仓库公开并使用 Apache License 2.0。
- 原因：允许学习和复用，同时保留明确版权和专利条款。
- 后果：必须严格隔离密钥、签名、真实数据和生产配置。

## ADR-005：后端保持可替换

- 状态：Accepted
- 决策：认证和远端数据库通过接口隔离，不让 Supabase/Firebase 类型进入领域层。
- 原因：主要使用地区和部署方式尚未最终确认。
- 后果：会增加少量适配层，但降低平台锁定风险。

## ADR-006：初始最低 Android 版本

- 状态：Superseded by ADR-009
- 决策：建议 `minSdk 26`。
- 原因：简化日期 API 和候选后端 SDK 集成，同时覆盖常用现代设备。
- 确认条件：检查实际测试设备和目标用户设备范围。

## ADR-007：云端方案

- 状态：Proposed
- 候选：Supabase/PostgreSQL、Firebase、自建 API + PostgreSQL。
- 建议：优先验证 Supabase/PostgreSQL 或可自托管方案；主要在中国大陆使用时必须先实测网络和部署区域。
- 确认条件：地区、成本、运维接受程度和端到端加密需求明确。

## ADR-008：端到端加密

- 状态：Proposed
- 决策问题：MVP 是否要求服务端无法读取记录内容。
- 影响：密钥恢复、多设备加入、导入导出和搜索方案都会改变。
- 确认条件：在实现云端 schema 前决定。

## ADR-009：Android 工程标识

- 状态：Accepted
- 决策：技术项目名使用 `DailyRecord`，Android `namespace` 与 `applicationId` 使用 `io.github.litaog.dailyrecord`，初始 `minSdk` 使用 26。
- 原因：包名与 GitHub 所有者命名空间一致且全局辨识度高；技术标识不绑定后续可调整的产品展示名称。
- 后果：发布到应用商店后不再修改 `applicationId`；展示名称可通过资源和本地化独立演进。

## ADR-010：记录状态与数量分离

- 状态：Accepted
- 决策：每日活动记录显式区分 `UNSET`、`DONE`、`MISSED`、`SKIPPED`；次数作为独立数值。次数改为 0 时转为 `MISSED`，清除时回到 `UNSET`。
- 原因：没有记录不等于明确没做；开源产品的真实 Issue 证明把数值与状态分开处理会造成错误统计与误导性日历。
- 后果：数据库、UI、统计和同步都必须保留状态语义，不能只依赖 quantity 推断所有情况。

## ADR-011：本地产品闭环先于账户同步

- 状态：Accepted
- 决策：P0 只要求离线记录、日历、活动管理和可信统计；账户与跨设备同步进入 P2。
- 原因：登录和网络不能阻塞最核心的每日记录价值，也应先稳定数据模型再定义冲突策略。
- 后果：游客数据从第一天就是正式数据；未来登录必须支持安全绑定和可解释合并。

## ADR-012：活动身份色与记录状态分离

- 状态：Accepted
- 决策：`Activity.color` 只表示活动身份并跨页面保持一致；`RecordStatus` 使用图标、文字、边框、纹理和可访问描述表达，不创建状态专属活动色。空日期默认不显示 `0` 汇总胶囊。
- 原因：同一颜色不能同时可靠表达活动、状态和强度；当前视觉原型已经暴露跨屏手冲颜色不一致和标记系统语义冲突。
- 后果：Figma 与 Compose 实现前必须修订标记系统；单活动强度只能在不破坏状态可读性的前提下使用同一活动色的层级变化。
