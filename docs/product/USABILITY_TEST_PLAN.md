# 5 人可用性测试计划

状态：Ready to run after interactive prototype
更新时间：2026-07-16

本计划落实 GitHub Issue #8。测试目标不是让参与者评价视觉风格，而是验证记录速度、状态理解、日历扩展性和统计可信度。

## 1. 前置条件

- 可点击原型或 Android 构建覆盖月历、日期详情、统计和活动管理。
- 测试数据同时包含 0、2、4、8+ 项活动，以及 `UNSET/DONE/MISSED/SKIPPED`。
- 手冲在所有页面使用同一飞机图标和同一活动基础色。
- 研究主持人不解释正确路径，除非参与者明确放弃。

## 2. 参与者

至少 5 名目标用户：Android 主力用户，过去三个月有个人记录行为；至少 3 人同时记录次数型和完成型活动。尽量包含一名大字号或 TalkBack 使用者；若无法招募，另安排可访问性专项测试，不能把普通用户测试当作替代。

## 3. 任务与成功条件

| 任务 | 起始状态 | 成功条件 | 关键指标 |
|---|---|---|---|
| T1 找出某天活动数量 | 月历全部活动 | 正确解释 2、4、8+ 胶囊的总数 | 正确率、耗时、误读 |
| T2 今天手冲从 1 改为 2 | 今天详情关闭 | 完成修改且知道已保存 | 点击数、用时、是否误触 |
| T3 健身标记“没做” | 日期详情 | 选中 `MISSED` 并解释与未记录的区别 | 状态理解率 |
| T4 将未来计划日标记“跳过” | 未来日期详情 | 选中 `SKIPPED`，不把它解释为失败 | 状态理解率 |
| T5 精确输入阅读 35 分钟 | DURATION 行 | 找到精确输入并保存 35 | 入口发现率、用时 |
| T6 筛选手冲 | 月历全部活动 | 识别飞机、状态和数量；不把筛选当时间范围 | 正确率 |
| T7 解释年统计 | 手冲年统计 | 解释 128 次、74 天、日均 1.7 和未来空月份 | 统计理解率 |
| T8 新增并排序第 7 个活动 | 活动管理 | 创建通用活动并调整顺序 | 完成率、错误数 |
| T9 归档后找历史 | 活动管理 | 归档后能从历史统计找到记录 | 完成率、信心 |
| T10 误操作后恢复 | 任意记录成功后 | 使用撤销或清除恢复原状态 | 撤销发现率 |

T2 是“常用记录”计时任务：从看到月历开始，到界面确认本地保存结束。目标中位数不超过 5 秒、路径不超过 3 次主要点击。

## 4. 主持词

- “请像平时一样完成任务，过程中说出你在找什么。”
- “我测试的是产品，不是你；找不到也很有价值。”
- 遇到停顿先问“你现在预期会发生什么”，不立即提示控件。
- 完成后问“你怎么知道已经保存/完成/跳过？”

## 6. 问题严重度

| 等级 | 定义 | 处理方式 |
|---|---|---|
| S0 阻断 | 数据丢失、错误统计、无法完成核心任务 | 暂停对应实现/发布，立即修复 |
| S1 严重 | 多人失败或形成错误理解，且无明显恢复路径 | 进入下个迭代前修复 |
| S2 中等 | 明显变慢、需要提示或造成低信心 | 排入 P0/P1 并验证修复 |
| S3 轻微 | 文案、视觉或偏好问题，不影响任务 | 记录并批量优化 |

同一问题出现在 2 人及以上时至少为 S1 候选；单个问题如果会造成数据错误，也可直接定为 S0/S1。

## 7. 通过门槛

- 核心任务完成率 ≥ 90%。
- T2 中位用时 ≤ 5 秒，主要点击 ≤ 3。
- 4/8+ 活动总数识别正确率 ≥ 90%。
- 单活动筛选图标、状态和数量识别正确率 ≥ 90%。
- 统计口径理解率 ≥ 80%。
- `UNSET/MISSED/SKIPPED` 没有导致数据语义错误的 S0/S1 问题。
- 所有失败都有设计决策：修复、延期、拒绝或需要更多研究。

样本只有 5 人，比例用于暴露问题和比较任务，不用于推断总体市场比例。Beta 后必须用真实使用数据重新校准。

## 8. 测试后输出

1. 每项任务结果和关键片段摘要。
2. S0-S3 问题清单，关联截图/流程和责任 Issue。
3. 修复前后假设以及需要复测的任务。
4. 对 Product Strategy、KPI、Backlog 和验收标准的修改。
5. Issue #8 回填结果；未达门槛时 PR #3 保持 Draft。
