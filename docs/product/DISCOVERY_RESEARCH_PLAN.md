# 发现研究计划

状态：Ready to recruit
更新时间：2026-07-16

本计划验证“是否值得做”和“为谁做”，与[可用性测试计划](USABILITY_TEST_PLAN.md)的“能否顺利使用”分开。当前开源 Issue 只能证明相似产品存在痛点，不能替代 Daily Record 自己的目标用户证据。

## 1. 需要验证的假设

| 编号 | 假设 | 反证信号 | 影响 |
|---|---|---|---|
| H1 | 用户确实需要在同一工具中记录次数型和完成型活动 | 大多数人只需要单一习惯勾选或专业健身工具 | 收窄目标用户或模板范围 |
| H2 | 月历比列表更适合回看“哪天发生了什么” | 用户主要关心连续目标或今天待办，不回看日期 | 重估月历默认入口 |
| H3 | `UNSET/MISSED/SKIPPED` 的区别对用户有价值 | 用户认为四状态过重或无法稳定解释 | 简化默认交互，保留数据层语义 |
| H4 | 可信精确表格比复杂图表更重要 | 用户只看趋势，不会核对明细 | 调整统计信息层级 |
| H5 | 无强制登录和本地优先能增加信任与首次使用 | 用户更在意自动云备份，担心本地丢失 | 提前验证导出/恢复，不强制提前上云 |
| H6 | 通用活动扩展是长期价值，不会使首屏过载 | 用户只使用 1-2 项且不创建自定义活动 | 降低活动管理曝光，保留底层扩展性 |

## 2. 参与者

第一轮至少 5 人，尽量不要与后续可用性测试完全重合：

- 2 名当前使用日历、备忘录或表格记录个人行为的用户。
- 2 名使用习惯/健身/健康记录应用的用户。
- 1 名曾因复杂、广告、登录或数据不可信而放弃记录应用的用户。
- 至少 3 人同时存在次数型和完成型记录需求。
- Android 为主要手机；年龄、技术熟练度和记录频率不要全部相同。

排除：项目作者、直接参与设计的人、只为奖励而参加且没有相关经历的人。

## 3. 访谈流程（30-40 分钟）

1. 最近一次记录某项个人活动是什么时候？请按当时实际过程讲一遍。
2. 为什么记录？之后在什么时候回看？回看后做了什么决定？
3. 目前用什么工具？最麻烦、最容易忘记或最不可信的部分是什么？
4. 有没有同时记录“次数”和“做没做”的场景？它们为什么不能用同一种方式？
5. “没记录”“明确没做”“主动跳过”在你的真实生活中是否不同？请举例。
6. 如果手机换机、离线或应用停止维护，你最担心什么？
7. 展示价值主张，不展示完整 UI：哪一部分有用，哪一部分多余？
8. 如果下周不能使用这个产品，你会继续用什么替代？

禁止诱导：不要先解释四状态的正确答案，不要问“你喜欢这个功能吗”，不要把未来承诺当成现有能力。

## 4. 记录模板

每位参与者只记录匿名编号：

| 字段 | 内容 |
|---|---|
| 参与者编号 | P01-P05 |
| 当前替代方案 | 应用类别或纸笔，不记录账号 |
| 主要记录任务 | 次数 / 完成 / 时长 / 混合 |
| 触发场景 | 何时想记录 |
| 回看决策 | 为什么查看历史 |
| 最高摩擦 | 原话摘要，不包含私人活动细节 |
| 信任顾虑 | 数据、隐私、同步、统计或其他 |
| H1-H6 证据 | 支持 / 反证 / 不确定 |
| 新问题 | 需要下一轮验证 |

不得保存真实活动备注、健康数据、联系方式、录音文件或可识别截图到公开仓库。

## 5. 决策规则

- 若少于 3/5 用户存在混合计量需求，保留通用模型，但首发定位收窄到最强场景。
- 若少于 3/5 用户会回看具体日期，重新评估月历是否应为默认入口。
- 若超过 2/5 无法自然区分 `MISSED` 与 `SKIPPED`，默认 UI 收起低频状态并通过二级操作提供。
- 若至少 3/5 把换机/丢失列为首要顾虑，P1 导出恢复优先于趋势图和提醒。
- 若精确统计无法驱动任何真实回顾决策，减少统计页面范围，不用更多图表制造价值感。

研究结论必须包含反证和未采纳理由；不得只挑支持当前方案的访谈片段。

## 6. 输出

- 一页研究摘要：核心问题、用户任务、替代方案和高风险假设。
- H1-H6 证据矩阵与置信度。
- 对 P0 范围、信息架构和 Backlog 的具体变更。
- 未解决问题与下一轮研究计划。
