# 版本路线图

路线图使用可验收阶段，而不是固定日期。每个阶段结束时应产生一个可验证结果，并通过 Pull Request 合并。

## Phase 0：需求与开发环境基线

目标：在写业务代码前消除会影响工程结构的关键不确定性。

- 确认应用名称、包名、`minSdk` 和目标平台。
- 检查 Android Studio、Android SDK、内置 JDK、Git 和真机/模拟器。
- 确认主要使用地区与后端候选。
- 确认统计口径和颜色规则。
- 建立 GitHub Issues、Milestones 和基础项目看板。

完成条件：关键决策写入 `DECISIONS.md`，开发机能成功构建官方空白 Compose 项目。

当前进度（2026-07-16）：Android 骨架、CI、产品策略、开源调研、信息架构、KPI 和五张视觉基线已完成。Phase 0 处于验证阶段：先关闭原型颜色语义设计债并执行 5 人可用性测试；Figma 教育权限不阻塞 #5 数据模型准备。

## Phase 1：Android 工程骨架

目标：建立可以长期扩展的最小工程。

- 创建 Kotlin + Compose + Material 3 工程。
- 配置版本目录、代码格式、静态检查和 CI。
- 建立设计系统、导航、依赖注入和错误处理基础。
- 建立 `core`、`feature` 和 `sync` 的边界，早期按需要物理拆分模块。
- 添加单元测试和 Compose 测试样例。

完成条件：Debug APK 可安装；CI 能执行编译、lint 和测试。

## Phase 2：本地 MVP

目标：不登录、不联网也能完成核心记录。

- Room schema、DAO、Repository 和迁移测试。
- 默认活动模板与活动管理。
- 月历、日期详情和快速记录。
- 次数型与完成型活动。
- 时长型活动的 5 分钟步进和精确输入。
- 归档、备注和用户设置。
- 周/月/年/全部历史的精确统计表。

完成条件：飞行模式下完成完整记录流程，杀死进程和重启后数据不丢失。

## Phase 3：统计与可视化

目标：在 P0 精确统计之上增加可解释的高级可视化。

- 柱状图/折线图和年度热力图。
- 空数据、部分周期和跨年边界。
- 统计查询性能测试与索引。

完成条件：高级图表与 P0 精确表使用同一查询结果，固定测试数据集一致且图表具备可访问摘要。

## Phase 4：账户与云同步

目标：让用户在其他设备登录后恢复并继续使用原数据。

- 选定后端并建立开发、测试、生产环境。
- 邮箱身份、会话恢复和游客数据绑定。
- Outbox 同步队列、WorkManager 重试和同步状态。
- 新增、修改、归档、软删除的双向同步。
- 多设备同日编辑冲突测试。
- 行级权限和账户数据删除。

完成条件：两台设备在离线编辑后恢复联网，最终得到可解释且一致的数据。

## Phase 5：隐私、备份与可靠性

- 应用锁和生物识别。
- CSV/JSON 导出与导入验证。
- 崩溃恢复、数据库备份和迁移演练。
- 日志脱敏和安全审查。
- 无障碍、深色模式和大字号检查。

完成条件：安全清单通过，导出的数据可在干净安装中恢复。

## Phase 6：Beta 与发布

- 应用图标、截图、商店说明和隐私政策。
- Release 签名和 GitHub Secrets 配置。
- Baseline Profile、性能和电量检查。
- 内部测试、封闭测试和反馈处理。
- 生成 AAB 并准备发布渠道。

完成条件：发布候选版本可复现构建，阻断级问题为零。

## 后续候选

- 提醒和桌面小组件。
- 目标、连续天数和最长连续记录。
- 健身时长、训练类型、动作和重量。
- Health Connect。
- 可选端到端加密。
- 多语言、平板自适应布局。
- iOS 或 Web；启动前需要重新评估 KMP/共享领域层。

## 迭代规则

- 每个 Phase 可以拆分多个小 PR，但一个 PR 只交付一个清晰主题。
- 不以“功能看起来完成”作为结束，必须满足测试和数据迁移要求。
- 未完成的后端能力不能阻塞本地核心记录。
- 新功能先扩展通用模型，确实无法表达时再新增专用领域模型。
