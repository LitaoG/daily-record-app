# AGENTS.md

本文件为参与本仓库工作的自动化代理和开发者提供长期有效的项目约束。

## 开始工作前

1. 阅读 `README.md`、`docs/PRODUCT.md`、`docs/ARCHITECTURE.md` 和 `docs/DECISIONS.md`。
2. 检查当前分支和工作区状态，保护用户已有改动。
3. 新需求先判断是否应通过通用活动模型实现，避免为“手冲”或“健身”复制一套独立数据层。
4. 涉及统计口径时，以 `docs/STATISTICS.md` 为准；如果文档没有定义，先记录决策再实现。
5. 涉及账号、同步、数据库迁移和隐私时，以兼容旧数据为优先。

## 工程原则

- 本地优先：核心记录功能在无网络时必须可用。
- 单一事实来源：界面只从本地 Room 数据库读取应用数据。
- 分层：UI 不直接调用 Room、网络 SDK 或认证 SDK。
- 可替换后端：业务层不能依赖具体云服务的数据类型。
- 可迁移：所有 Room schema 变更必须提供迁移与测试，不允许以破坏性迁移作为默认方案。
- 通用建模：活动名称、颜色、图标、计量方式均由配置驱动。
- 隐私最小化：不采集实现核心功能不需要的数据。
- 可验证：新增统计和日期逻辑必须包含单元测试。

## Git 与 Pull Request

- `main` 必须始终可构建。
- 使用短生命周期功能分支，代理分支使用 `agent/<description>`。
- 提交中不得包含密钥、签名文件、真实用户数据、数据库文件、APK 或 AAB。
- PR 必须写明变更、原因、用户影响和验证方式。
- 一个 PR 尽量解决一个主题，避免顺手重构无关模块。

## 完成标准

- 需求与验收条件明确。
- 代码符合现有架构边界。
- 相关测试通过。
- 数据库变化具备迁移测试。
- 文档与代码一致。
- 不引入新的公开仓库敏感信息。
